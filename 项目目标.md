### boss直聘自动化求职助手

技术栈：FastAPI + Playwright + SQLite + OpenAI API

目标：能够自动化地在 boss直聘网站上完成职位抓取 -> 职位匹配 -> 触发职位“立即沟通” -> 打开消息页面 -> 找到刚刚的职位 -> 发送 AI 生成的消息 -> 关闭除了主页之外的所有页面。

发送的 AI 消息应该包括以下内容：

1. 问候
2. 说明本条消息来源
3. AI 岗位匹配度意见
4. 期待进一步沟通

## 已确认决策

1. 大模型使用 OpenAI 兼容接口。
2. OpenAI 配置从 `.env` 读取：
   - `OPENAI_BASE_URL`
   - `OPENAI_API_KEY`
   - `OPENAI_MODEL`
3. 期望薪资按“最低值”判断，岗位薪资低于期望最低值则跳过。
4. 期望城市允许填写多个候选城市，命中任一候选城市即可。
5. 是否触发“立即沟通”由大模型判断，不使用固定分数阈值。
6. 暂不需要排除外包、培训、驻场等岗位类型。
7. 用户信息初版只保留：
   - 期望薪资
   - 期望城市
   - 个人描述
8. AI 消息完全自动发送，不需要人工确认。
9. boss直聘每天最多沟通 150 个岗位，系统需要按此设置每日上限。
10. 前端技术方案可按实现便利性选择，优先完成可用的 Web 管理页面。
11. CDP 使用 Microsoft Edge，不使用 Chrome。
12. 实战测试允许真实点击和真实发送消息。
13. 测试阶段不额外设置沟通上限，但每一步必须确认效果符合预期后再进入下一步开发。
14. 明确的风控表现：
    - 操作后自动回到首页
    - 页面被关闭
15. 发送消息可以说明是 AI 辅助匹配，因为本项目用于寻找 AI 应用开发、AI Agent 开发相关岗位。

## 默认用户信息

1. 期望薪资：20k 以上。
2. 候选城市：上海。
3. 求职方向：AI 应用开发工程师、AI Agent 开发工程师、RAG / GraphRAG 开发、智能体工程化、AI 后端开发。
4. 个人描述：
   - 具备近 2 年 AI 应用开发与项目交付经验，主要方向包括企业级 AI 应用平台、智能客服、智能 BI、招聘助手、RAG / GraphRAG 和 Agent 工作流。
   - 熟悉 Python、FastAPI、RESTful API、接口联调、异常处理、日志排查和后端工程化，能够将 AI 能力封装为稳定可交付的服务。
   - 熟悉 LangGraph、LangChain、Dify 等智能体与工作流开发方式，理解结构化输出、Function Call、MCP 工具接入、多步骤任务处理、HIL、记忆机制和上下文管理。
   - 熟悉 RAG / GraphRAG 核心链路，包括数据清洗、文档解析、文本切分、Embedding、向量检索、全文检索、混合检索、RRF 排序融合、知识溯源、图谱增强检索和幻觉评估。
   - 熟悉 PostgreSQL、MySQL、Redis、Elasticsearch、Weaviate、Milvus、Neo4j 等存储和检索组件在知识库、向量检索、图谱查询和业务系统中的应用。
   - 有企业级 AI 应用平台 RAG 检索引擎、插件化工具调用模块、GraphRAG 智能客服、Text-to-SQL / Text-to-Cypher 智能查询、智能 BI 报表助手和招聘助手等项目经验。
   - 重视工程落地、结果可复核、过程可观测、异常兜底和交付稳定性，能够从需求拆解、方案设计、开发联调到上线验证推进完整链路。

## 项目 Web 页面

1. 用户信息填写：
   - 期望薪资
   - 期望城市，支持多个候选城市
   - 个人描述
2. 打开项目的 boss直聘页面登录，并保存登录状态。
3. 开始按钮：
   - 职位抓取
   - 职位匹配
   - 触发职位“立即沟通”
   - 发送 AI 生成的消息
4. 停止按钮。
5. 已经沟通的岗位信息查看，支持分页查询。
6. 运行日志和异常记录查看。

## CDP 与实战测试要求

1. 浏览器调试使用 Microsoft Edge CDP。
2. 项目可以连接本机已经登录的 Edge 调试端口。
3. CDP 仅用于调试、页面观测、页面控制和复现真实用户路径。
4. 不使用 CDP 实现隐藏自动化特征、验证码绕过、风控规避等对抗性能力。
5. 测试必须尽可能采用 debug/headed 模式，确保可以观察每一步页面变化。
6. 测试时允许真实点击“立即沟通”。
7. 测试时允许真实发送 AI 生成的消息。
8. 测试阶段不额外设置沟通数量上限，但需要遵守 boss直聘每日 150 次沟通限制。
9. 每一个关键动作执行后，都必须确认页面结果符合预期，再进入下一步。
10. 如果观察到动作成功，应继续推进下一步开发；如果观察到异常，应记录并暂停该路径。
11. 每次实战测试都要记录：
    - 测试时间
    - Edge CDP 连接方式
    - 起始 URL
    - 操作步骤
    - 每一步操作前后的 URL 和标题
    - 关键元素是否存在
    - 截图或 trace 路径
    - 是否出现风控表现
    - 最终结果

## 开发原则

1. boss直聘有严格的反爬取和反自动化机制。
2. 在使用自动化点击、跳转、输入之前，需要确认目标元素存在、可见、可操作。
3. 执行点击、跳转、输入之后，需要确认执行效果。
4. 如果触发 boss直聘的风控操作，例如页面退出、自动回到首页、登录失效、异常跳转等，需要记录事件，方便后续开发。
5. 需要探索 boss直聘页面结构，例如顶部菜单、搜索框、URL、消息页面、岗位详情页等，并留档做笔记。
6. 每次打开新的关键页面都需要记录页面 URL、标题、关键元素和观察结果。
7. 遇到验证码、登录失效、页面异常或疑似风控时，初版不做绕过处理，应该记录并暂停任务，等待用户处理。
8. 如果某种 Playwright 操作方式容易触发异常，可以改用更接近普通用户的操作方式，例如鼠标点击、键盘输入、等待页面稳定等。
9. 不实现验证码绕过、隐藏自动化特征、规避平台风控等对抗性能力。
10. 如果操作后自动回到首页，视为触发风控，需要立即记录并暂停当前流程。
11. 如果操作后页面被关闭，视为触发风控，需要立即记录并暂停当前流程。

## 页面探索与留档要求

探索 boss直聘页面时，需要先形成页面地图，再沉淀可执行路径。探索结果必须留档，方便下一次开发继续使用。

建议文档：

1. `docs/boss-pages.md`：页面结构笔记。
2. `docs/boss-flow.md`：可行操作路径。
3. `docs/boss-selectors.md`：选择器候选、稳定性说明和备选方案。
4. `docs/boss-test-runs.md`：实战测试记录。

建议运行产物目录：

1. `runtime/screenshots/`：关键步骤截图。
2. `runtime/traces/`：Playwright trace/debug 记录。
3. `runtime/logs/`：结构化运行日志。

页面探索流程：

1. 使用 Edge CDP 连接已登录浏览器。
2. 打开 boss直聘推荐岗位页。
3. 记录页面 URL、标题、登录态、主要区域和关键按钮。
4. 识别城市切换入口、岗位列表、岗位详情、薪资、城市、JD、“立即沟通”按钮。
5. 每次点击或跳转前，记录当前状态和预期结果。
6. 每次点击或跳转后，记录实际结果，并判断是否符合预期。
7. 对每个关键元素记录多个选择器候选，例如文本定位、role 定位、CSS 定位、URL 或标题辅助判断。
8. 标注每个选择器的稳定性和风险。
9. 形成从推荐页到消息发送完成的完整路径。
10. 如果出现回首页、页面被关闭、登录失效、验证码、页面结构异常等情况，记录为异常路径。

可行路径文档应包含：

1. 前置条件。
2. 起始页面。
3. 操作步骤。
4. 每一步的目标元素。
5. 每一步的成功判定。
6. 每一步的失败判定。
7. 页面截图或 trace 证据。
8. 推荐选择器。
9. 备选选择器。
10. 需要人工介入的情况。

## 大致流程

1. 确保登录状态之后，打开 `https://www.zhipin.com/web/geek/jobs`。
2. 该页面为个性化岗位推荐页面，页面内可能已有 boss直聘侧填写的多个期望岗位。
3. 期望岗位名称可能变化，不能把流程绑定到固定岗位名称，例如“高性能计算工程师”“大模型算法”“算法工程师”只能作为当前页面样本。
4. 需要通过页面结构、位置、HTML 元素关系识别“期望岗位/求职意向”区域，并遍历该区域下的岗位项。
5. 期望岗位不一定对应程序填写的候选城市，不能假设某个期望岗位一定是上海。
6. 打开页面后，优先读取页面内已有期望岗位项及其对应城市。
7. 如果任一期望岗位项对应程序候选城市，则切换到该期望岗位项，并将切换后的推荐页作为主页。
8. 如果页面内已有期望岗位项都不对应程序候选城市，则需要切换页面城市到程序候选城市。
9. 确保城市正确后，在主页点击第一个推荐岗位，获取 JD、薪资、城市、公司和岗位 URL。
10. 如果薪资低于用户期望最低值，则记录并跳过。
11. 如果岗位城市不在用户候选城市内，则记录并跳过。
12. 如果薪资和城市符合预期，则使用大模型对 JD 和用户个人描述进行匹配判断。
13. 如果大模型认为岗位不匹配，则记录并跳过。
14. 如果大模型认为岗位匹配，则触发该岗位的“立即沟通”按钮。
15. 触发“立即沟通”后，打开消息页面。
16. 在消息页面找到刚刚沟通的岗位。
17. 发送 AI 生成的消息。
18. 保存已沟通岗位的信息和发送内容。
19. 关闭除了主页之外的所有页面。
20. 打开下一个推荐岗位，继续重复，直到用户停止、达到每日上限、登录失效或发生需要人工处理的异常。

## OpenAI 使用要求

岗位匹配时，大模型输入应包含：

1. 用户期望薪资
2. 用户候选城市
3. 用户个人描述
4. 岗位名称
5. 公司名称
6. 岗位薪资
7. 岗位城市
8. JD 原文

岗位匹配输出应为结构化结果，至少包含：

1. `matched`：布尔值，表示是否匹配
2. `reason`：简短理由

消息生成输入应包含：

1. 用户个人描述
2. 岗位名称
3. 公司名称
4. JD 原文
5. AI 匹配理由

消息生成结果应满足：

1. 语气自然、简洁、礼貌。
2. 包含问候。
3. 说明本条消息来源于 AI 对岗位信息和个人情况的匹配判断。
4. 表达岗位匹配度意见。
5. 表达期待进一步沟通。
6. 避免过长，适合 boss直聘聊天窗口首条消息。
7. 可以结合用户正在寻找 AI 应用开发、AI Agent 开发相关岗位的背景来表达匹配点。
